# SPEC: Filesystem Project Pivot

**Version:** 7.0.0  
**Date:** 2026-04-24  
**Status:** Implemented baseline

## Summary

Book Editor has been pivoted from the legacy `book + chapters[]` JSON model to a filesystem-first project model driven by:

- `backend/data/projects/<projectId>/book-editor.manifest.json`
- `backend/data/projects/<projectId>/content/<nodeId>.md`
- `backend/data/projects/<projectId>/assets/`
- `backend/data/projects/<projectId>/.book-editor/`

The primary UX is now a modular thesis workspace with:

- arbitrary-depth outline
- manifest-first ordering
- per-node markdown files
- inline multi-section canvas
- per-project writing rules
- AI revise sidebar
- backend-managed export

## Data Model

### Manifest

`book-editor.manifest.json`

```ts
interface ProjectManifest {
  id: string
  title: string
  slug: string
  rootNodeIds: string[]
  nodes: Record<string, ProjectNode>
  rules: ProjectRuleSet
  exportProfiles: {
    defaultDocxTemplate?: string
    defaultPdfEngine?: string
  }
  createdAt: string
  updatedAt: string
}
```

### Node

```ts
interface ProjectNode {
  id: string
  title: string
  slug: string
  parentId: string | null
  children: string[]
  contentFile: string
  collapsed: boolean
  createdAt: string
  updatedAt: string
}
```

### Rules

```ts
interface ProjectRuleSet {
  mustFollow: string[]
  mustNotFollow: string[]
  terminology: string[]
  memoryNotes: string[]
  styleProfile: string
}
```

## Backend

- `GET /api/projects`
- `POST /api/projects`
- `GET /api/projects/:projectId/tree`

- `PUT /api/projects/:projectId/nodes/:nodeId/content`

- `PATCH /api/projects/:projectId/nodes/:nodeId/move`
- `DELETE /api/projects/:projectId/nodes/:nodeId`
- `PATCH /api/projects/:projectId/rules`
- `POST /api/projects/:projectId/ai/revise`
- `GET /api/projects/:projectId/markdown`
- `POST /api/projects/:projectId/export/:format`

### Behavior

- Project creation creates one initial root node plus empty content/rules/export folders.
- Node ordering and hierarchy are controlled only by manifest arrays, not by filenames.
- Node saves write markdown to `content/<nodeId>.md` and snapshot the manifest into `.book-editor/versions/`.
- AI revise uses the configured default provider in backend, with project rules and node ancestry included in prompt context.
- Export formats:
  - `markdown`: merged manuscript in manifest order
  - `zip`: archive of the project folder via `Compress-Archive`
  - `docx`: Pandoc conversion from merged markdown
  - `pdf`: Pandoc conversion from merged markdown

### Runtime Dependencies

- `pandoc` must be available in PATH for DOCX and PDF export.
- PowerShell `Compress-Archive` is used for ZIP export.
- AI revise requires at least one enabled provider in `/api/providers`.
- Frontend reads `VITE_API_BASE_URL` from `frontend/.env`.

## Frontend

### Routes

- `/`
- `/projects/:projectId`

### Main UX

- Home page lists filesystem projects and creates new projects.
- Editor page renders:
  - recursive left outline
  - multi-section document canvas in the middle
  - rules + AI Wingman sidebar on the right
- Each section card contains:
  - editable markdown textarea
  - rendered preview beside it
- Autosave is debounced per node and flushed on blur.
- Tree supports add, rename, delete, expand/collapse, and drag-drop reorder.
- AI sidebar supports:
  - preset revise commands
  - custom instruction
  - replace current selection/full node
  - insert AI result below selection/current node

## Accepted Constraints

- Project root is currently managed inside `backend/data/projects`, not arbitrary user-chosen folders.
- Section editing currently uses textarea-based editing in the pivot baseline, not CodeMirror blocks.
- Long document UX is implemented as a stacked multi-node canvas; no dedicated virtualization library is used yet.
- Rules are currently project-level only in shipped UI.
